## 1. Overview

**Description**: A shared ingredient master list stores per-100-unit nutrient
values for each ingredient. From it the app computes a recipe's aggregated
nutrients (and per-portion values), and exposes ingredient management so the
catalogue can grow. The nutrient aggregates also feed cycle scoring (REQ-011).

**Business Value**:
- Gives every recipe a nutrient profile without any external API
- A shared catalogue avoids re-entering nutrient data per recipe

## 2. Functional Requirements

### FR-1201: Ingredient master catalogue with nutrient fields

- **Description**: Each `ingredients_master` row carries a name, category,
  base unit/size, and ~15 nutrient fields (kcal, protein, carbohydrates, fat,
  sugar, fiber, salt, sodium, calcium, iron, magnesium, zinc, vitamin_d,
  vitamin_b6, vitamin_b12, vitamin_e).
- **Acceptance Criteria**:
  - AC-012-01: An authenticated `POST /api/ingredients-master` with a name and
    nutrient values creates an ingredient (HTTP 201) persisting those fields
  - AC-012-02: A duplicate name is rejected (HTTP 409)
  - AC-012-03: `PUT` updates and `DELETE` removes an ingredient (owner-restricted
    mutations; reads are public)

### FR-1202: List ingredients with nutrients

- **Description**: Ingredients are listable for the catalogue/filter UIs.
- **Acceptance Criteria**:
  - AC-012-04: `GET /api/nutrition/ingredients` returns HTTP 200 with ingredients
    ordered by category, then name (see REQ-008)
  - AC-012-05: An empty catalogue returns HTTP 200 with an empty list

### FR-1203: Per-recipe nutrient calculation

- **Description**: A recipe's nutrients are the sum of each ingredient's
  contribution, scaled by `quantity / base_size`; per-portion values divide the
  total by the portion count.
- **Acceptance Criteria**:
  - AC-012-06: For a single ingredient, the total nutrient equals
    `(quantity / base_size) × ingredient_nutrient`
  - AC-012-07: Multiple ingredients sum correctly across mixed units
  - AC-012-08: Per-portion values equal `total / portions`, normalised to 2
    decimals; zero/negative portions are rejected
  - AC-012-09: `salt` is included in the calculated totals (distinct from sodium)

### FR-1204: Unit-to-base amount conversion for calculation

- **Acceptance Criteria**:
  - AC-012-10: `convertToBaseAmount` converts grams directly and pieces via the
    ingredient's per-piece weight; an unknown unit throws a clear error

### FR-1205: Quality gates

- **Acceptance Criteria**:
  - AC-012-11: `npm run lint`, `npx tsc --noEmit`, and `npm test` all pass

## 3. Non-Functional Requirements

- **Storage**: `ingredients_master` (migration `005_create_ingredients_master.sql`;
  `salt` added by `004_add_salt_field.sql`) and `recipe_nutrients`
  (migration `002_create_nutrition_tables.sql`). SQLite-only (ADR-008).
- **Base-unit semantics**: nutrient values are per `base_size` of `base_unit`
  (default 100 g; 100 ml for liquids).
- **Matching**: recipe ingredients join the master list on
  `LOWER(TRIM(name))` (case/whitespace-insensitive).
- **Precision**: calculated values normalised to 2 decimals
  (`normalizeNutrientValue`).
- **Related**: REQ-008 (endpoint bug-fix), REQ-011 (consumes aggregates),
  REQ-013 (unit conversion — a separate, richer conversion subsystem).
