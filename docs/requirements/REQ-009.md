# REQ-009 — Recipe Photo

**Version**: 1.0
**Date**: 2026-06-15
**Status**: draft
**Epic**: Phase 2 — Recipe Experience
**Priority**: P2 (Feature)
**Owner**: Matthias Bender

---

## 1. Overview

**Description**: A recipe can have a single photo. The photo is uploaded
on the recipe create page (`/recipes/new`) and the recipe edit page
(`/recipes/[id]/edit`). On the dashboard (`/dashboard`), the photo is
shown on each recipe card instead of the text description.

**Business Value**:
- Recipe cards become visual, making the dashboard easier to scan
- A photo communicates a dish faster than a text description

---

## 2. Functional Requirements

### FR-901: Upload a photo when creating a recipe

- **Description**: The create form offers a file picker for a recipe
  photo. The selected image is stored with the new recipe.
- **Acceptance Criteria**:
  - AC-009-01: A JPG or PNG image (≤ 5 MB) selected on `/recipes/new`
    is persisted and associated with the created recipe
  - AC-009-02: A file that is not JPG/PNG, or is larger than 5 MB, is
    rejected with a clear error and the recipe photo is not stored

### FR-902: Add or replace a photo when editing a recipe

- **Description**: The edit form shows the current photo (if any) and
  allows replacing it with a new upload.
- **Acceptance Criteria**:
  - AC-009-03: Uploading a new image on `/recipes/[id]/edit` replaces the
    recipe's existing photo
  - AC-009-04: Only the recipe owner can upload or replace the photo
    (non-owner receives HTTP 403)

### FR-903: Serve the photo

- **Description**: A stored recipe photo is retrievable via a stable URL.
- **Acceptance Criteria**:
  - AC-009-05: `GET /api/recipes/[id]/image` returns the stored image
    bytes with the correct `Content-Type` when a photo exists
  - AC-009-06: `GET /api/recipes/[id]/image` returns HTTP 404 when the
    recipe has no photo

### FR-904: Show the photo on the dashboard

- **Description**: Each recipe card on the dashboard shows the recipe
  photo in place of the description.
- **Acceptance Criteria**:
  - AC-009-07: A recipe card whose recipe has a photo renders the image
    instead of the description text
  - AC-009-08: A recipe card whose recipe has no photo falls back to the
    description text (existing behaviour, no regression)

### FR-905: Quality gates

- **Acceptance Criteria**:
  - AC-009-09: `npm run lint`, `npx tsc --noEmit`, and `npm test` all pass

## 3. Non-Functional Requirements

- **Storage**: Images are stored on the filesystem under
  `.data/uploads/recipes/` (per `src/config/upload.ts`); only the file
  name is stored in the database. No binary blobs in the DB.
- **Dual-database**: The new `image_path` column must work on both SQLite
  (production) and PostgreSQL (migration tooling) code paths.
- **Base path**: All client-side image URLs use the `apiUrl()` helper so
  they respect the `/rezepte` sub-path in production (MAINT-003).
- **Validation**: Reuse the existing `getValidationError()` from
  `src/config/upload.ts` (JPG/PNG, 5 MB limit).
