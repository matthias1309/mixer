## 1. Overview

**Description**: Recipe ingredient quantities can be converted between
measurement units (TL, EL, ml, l, g, kg, Stück, Prise), including volume↔weight
conversion via per-ingredient density. A recipe can be scaled to a new serving
count, with proportional quantities and automatic "unit promotion"
(e.g. 3 TL → 1 EL) and sensible rounding.

**Business Value**:
- Users can cook a recipe at any serving size without manual maths
- Ingredient amounts stay readable (promoted/rounded), not "6 TL"

## 2. Functional Requirements

### FR-1301: Same-category conversion

- **Acceptance Criteria**:
  - AC-013-01: Converting within volume (e.g. 1 TL → 5 ml) or weight
    (1000 g → 1 kg) returns the correct amount
  - AC-013-02: An unknown unit raises `UnknownUnitError` (HTTP 400 at the API)
  - AC-013-03: Two incompatible same-direction units with no path raise
    `ImpossibleConversionError`

### FR-1302: Cross-category conversion via density

- **Acceptance Criteria**:
  - AC-013-04: Volume→weight for an ingredient with density data converts
    correctly (e.g. 1 TL Mehl → 5 g)
  - AC-013-05: A cross-category conversion for an ingredient without density
    raises `MissingDensityError`

### FR-1303: Recipe scaling by serving count

- **Acceptance Criteria**:
  - AC-013-06: Scaling a recipe from N to M servings multiplies each ingredient
    quantity by `M/N`
  - AC-013-07: A scaled quantity is "promoted" to a larger unit when the
    threshold is reached (e.g. 6 TL → 2 EL; ml → l)
  - AC-013-08: Count/pinch units (Stück, Prise) are preserved and rounded to
    integers
  - AC-013-09: `newServings` outside 1–100 returns HTTP 400 and the recipe is
    unchanged (the scale endpoint is read-only — it returns scaled values,
    nothing is persisted)

### FR-1304: Quantity normalisation on ingredient add

- **Acceptance Criteria**:
  - AC-013-10: Adding a recipe ingredient stores both the original
    quantity/unit and a normalised quantity/unit in base units (ml or g); a
    non-normalisable unit stores `null` normalised fields

### FR-1305: Quality gates

- **Acceptance Criteria**:
  - AC-013-11: `npm run lint`, `npx tsc --noEmit`, and `npm test` all pass

## 3. Non-Functional Requirements

- **Architecture**: see **ADR-006** (DB-backed units/conversions/densities,
  load-once into in-memory maps, `UnitConverter` + `RecipeScaler` services).
- **Rounding** (`ROUNDING_RULES`): weight >50 g → nearest 5 g else 1 g; volume
  >100 ml → nearest 5 ml else 0.5 ml; TL/EL → 0.25 steps; count/pinch → integer.
- **Supported units**: TL, EL, ml, l, g, kg, Stück, Prise (extensible by seeding
  the `units`/`unit_conversions`/`ingredient_densities` tables).
- **Database**: SQLite-only (ADR-008). Conversion factors `NUMERIC(10,4)`,
  densities `NUMERIC(10,2)`.
