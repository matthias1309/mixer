# REQ-019 — Configurable Recipe Page Size

**Version**: 1.0
**Date**: 2026-06-26
**Status**: Implemented — documented retroactively
**Epic**: Phase 2 — Recipe Discovery
**Priority**: P3 (Enhancement)
**Owner**: Matthias Bender
**Traces**: REQ-015
**Traced by**: ARCH-019, TEST-019

> **Retroactive documentation.** The dashboard previously showed a fixed 10
> recipes per page before pagination kicked in (REQ-015, FR-1505). This story
> adds a control letting the user choose the page size. The underlying API
> already accepted a `pageSize` query parameter (clamped 1–100); this story
> wires that up to a UI control.

---

## 1. Overview

**Description**: The user can choose how many recipes are displayed per page
on the dashboard, instead of being fixed at 10.

**Business Value**:
- Users with many recipes or a large ingredient pantry can scan more results
  per page, reducing pagination clicks
- Users on smaller screens can keep the denser default

---

## 2. User Story

As a user browsing my recipes on the dashboard,
I want to choose how many recipes are shown per page,
in order to see more (or fewer) results at once without changing my filters.

---

## 3. Functional Requirements

### FR-1901: Page size selector

- **Description**: A dropdown next to the sort control lets the user pick a
  page size from a fixed set of options.
- **Acceptance Criteria**:
  - AC-019-01: The dashboard shows a page-size control with options 10, 20,
    50, 100, defaulting to 10
  - AC-019-02: Selecting a different page size re-fetches the recipe list
    using that page size

### FR-1902: Page reset on page-size change

- **Description**: Changing the page size while on a page other than 1 must
  not request an out-of-range page from the API.
- **Acceptance Criteria**:
  - AC-019-03: Changing the page size while on page > 1 resets the current
    page to 1

### FR-1903: API page-size parameter (pre-existing)

- **Description**: `GET /api/recipes?pageSize=<n>` accepts a page size,
  clamped to the range 1–100, defaulting to 10 when absent or invalid.
- **Acceptance Criteria**:
  - AC-019-04: `pageSize` values above 100 are clamped to 100
  - AC-019-05: `pageSize` values below 1 (or non-numeric) fall back to the
    default of 10
  - AC-019-06: The response's `totalPages` is computed using the requested
    `pageSize`

### FR-1904: Quality gates

- **Acceptance Criteria**:
  - AC-019-07: `npm run lint`, `npx tsc --noEmit`, and `npm test` all pass

---

## 4. Non-Functional Requirements

- **Scope boundary**: page size is a transient UI preference (component
  state), not persisted per user — out of scope for this story.
- **Database**: SQLite-only (ADR-008); no schema changes required.

---

## 5. As-Built Components

- `src/components/PageSizeDropdown.tsx` — the selector (10/20/50/100)
- `src/components/RecipeList.tsx` — accepts a `pageSize` prop, includes it in
  the API request, resets `page` to 1 on change
- `src/app/dashboard/page.tsx` — holds `pageSize` state, renders
  `PageSizeDropdown` next to `SortDropdown`
- `src/app/api/recipes/route.ts` (pre-existing, REQ-015) — parses and clamps
  `pageSize`

---

## 6. Out of Scope (without explicit discussion)

- Persisting the chosen page size across sessions (e.g. localStorage, user
  settings)
- Page sizes outside the 10/20/50/100 fixed set
