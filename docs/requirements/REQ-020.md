# REQ-020 — Absolute Session Lifetime

**Status:** draft
**Created:** 2026-07-02
**Traced by:** ARCH-020, TEST-020

## User Story

As a registered user,
I want my login session to expire at the latest 24 hours after I signed in — no matter how active I am,
in order to limit how long a stolen session token can be abused.

## Acceptance Criteria

### AC-020-01: Active session within the absolute lifetime keeps working

```gherkin
Given a user who logged in less than 24 hours ago
When they make an authenticated API request
Then the request succeeds
And a refreshed session token is issued (sliding window unchanged)
```

### AC-020-02: Session ends 24 hours after login despite continuous activity

```gherkin
Given a user who logged in more than 24 hours ago
And whose token has been refreshed continuously since then
When they make an authenticated API request
Then the request is treated as unauthenticated
And no refreshed session token is issued
```

### AC-020-03: Refreshed tokens preserve the original login time

```gherkin
Given a session token issued at login
When the token is refreshed during an authenticated request
Then the new token carries the original login timestamp
And the new token's remaining absolute lifetime is not extended
```

### AC-020-04: A fresh login restarts the absolute lifetime

```gherkin
Given a user whose previous session reached its absolute lifetime
When they log in again with valid credentials
Then a new session is created
And its absolute lifetime starts from the new login time
```

## Notes

- Security review finding 3 (2026-07-02): every API call issues a fresh 1h token
  (sliding window) with no absolute cap, so a stolen token can be kept alive
  indefinitely; logout only clears the cookie.
- Absolute lifetime: **24 hours**, sliding-window token expiry stays at 1 hour.
- Backwards compatibility: tokens issued before this change carry no original
  login time claim; their own `iat` is used as the login time so existing
  sessions age out naturally instead of being invalidated at deploy.
- `CLAUDE.md` documents the auth as "15-min access token + refresh rotation",
  which does not match the implemented 1h sliding window — align the docs as
  part of this story.
- Out of scope: server-side token revocation (deny list); may become a
  follow-up requirement.
