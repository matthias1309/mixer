# REQ-017 — REWE-Style Filter & Sort Engine

**Version**: 1.0
**Date**: 2026-06-25
**Status**: draft
**Epic**: Phase 3 — UX Overhaul (REWE-Style Redesign)
**Priority**: P2 (Feature)
**Owner**: Matthias Bender
**Traced by**: ARCH-017, TEST-017

> Part 3 of 4 of the REWE-style redesign. This story wires the metadata from
> REQ-016 into the **filter & sort engine** — API query params plus the sidebar
> filter groups — and adds a sort dropdown. It **composes with** the existing
> cycle-phase and ingredient filters, which must keep working unchanged.

---

## 1. Overview

**Description**: Extend `GET /api/recipes` and the `FilterPanel` so users can
narrow the list by effort, max total time, meal type, and tag groups
(Ernährung, Hauptzutat, Ernährungsform, Backen, Anlässe), and sort the results
(rating, time, newest). All filters are combinable with the existing
`phase` and `ingredients` filters.

**Business Value**:
- The core REWE browsing experience: stack several filters, see matching recipes
- Makes the REQ-016 metadata actually useful

---

## 2. Functional Requirements

### FR-1701: Extended query parameters

- **Acceptance Criteria**:
  - AC-017-01: `GET /api/recipes` accepts `difficulty`, `maxTime` (minutes),
    `mealType`, and `tags` (comma-separated) and returns only recipes matching
    **all** supplied criteria.
  - AC-017-02: Multiple `tags` are combined as **AND** (a recipe must carry every
    requested tag) — matching REWE's narrowing behaviour.
  - AC-017-03: The new filters compose with the existing `phase` and
    `ingredients` params (all applied together).
  - AC-017-04: An invalid/unknown filter value is ignored (treated as "not
    filtered") rather than erroring, so a stray query param never 500s.

### FR-1702: Sorting

- **Acceptance Criteria**:
  - AC-017-05: `GET /api/recipes` accepts `sort` ∈ { `rating`, `time`, `newest`,
    plus the existing options }, with a defined default, and orders results
    accordingly. (`rating` ordering uses REQ-018 data when available; until then
    it falls back to the default ordering.)

### FR-1703: Filter sidebar groups

- **Acceptance Criteria**:
  - AC-017-06: `FilterPanel` renders collapsible groups for Ernährung,
    Hauptzutat, Ernährungsform, Backen, Anlässe (multi-select), plus Aufwand and
    a max-time control, sourced from the REQ-016 vocabulary.
  - AC-017-07: The **cycle-phase group remains present and first** in the panel.
  - AC-017-08: Changing any filter updates the result list and the results
    counter; "Filter zurücksetzen" clears every group including the new ones.

### FR-1704: Sort control

- **Acceptance Criteria**:
  - AC-017-09: A sort dropdown above the grid lets the user pick the sort order
    and drives the `sort` query param.

### FR-1705: Quality gates

- **Acceptance Criteria**:
  - AC-017-10: `npm run lint`, `npx tsc --noEmit`, and `npm test` all pass.

---

## 3. Non-Functional Requirements

- **SQLite-only**; filtering happens in SQL (parameterised, no string-built
  predicates from user input).
- **Performance**: tag filtering uses the `recipe_tags(tag)` index from REQ-016;
  combined filters stay a single paginated query.
- **State**: active filters are reflected in the URL query string where practical
  so a filtered view is shareable/refresh-safe.
- **Base path**: client fetches use `apiUrl()` (MAINT-003).

## 4. Out of Scope

- Defining the metadata itself (REQ-016)
- Producing the rating values that `sort=rating` / `minRating` consume (REQ-018)
