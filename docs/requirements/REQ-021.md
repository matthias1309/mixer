# REQ-021 — Rate Limit for OCR Uploads

**Status:** draft
**Created:** 2026-07-02
**Traced by:** ARCH-021, TEST-021

## User Story

As the operator of the recipe app,
I want each user's recipe-photo OCR uploads to be rate limited,
in order to stop a single account from exhausting the server with
CPU-heavy text recognition (denial of service).

## Acceptance Criteria

### AC-021-01: Uploads within the limit are accepted

```gherkin
Given an authenticated user who has made fewer than 10 OCR uploads in the last 10 minutes
When they submit another recipe photo for OCR
Then the upload is accepted and processing starts
```

### AC-021-02: Uploads beyond the limit are rejected

```gherkin
Given an authenticated user who has made 10 OCR uploads in the last 10 minutes
When they submit another recipe photo for OCR
Then the request is rejected with HTTP 429
And the response includes a Retry-After header
```

### AC-021-03: Limits are per user

```gherkin
Given one user has reached the OCR upload limit
When a different authenticated user submits a recipe photo for OCR
Then that user's upload is accepted
```

## Notes

- Security review finding 4 (2026-07-02): `POST /api/recipes/ocr` ran a
  Tesseract OCR job (CPU-heavy, image up to 5 MB) per upload with no rate
  limit, so a single authenticated user could exhaust the server.
- Limit: **10 uploads per 10 minutes per user**, enforced via the existing
  in-memory rate limiter (`src/lib/auth/rateLimiter.ts`), keyed on the user id.
- Retrofit note: this requirement was written *after* the fix was implemented
  and reviewed (PR #40), to bring the change in line with the V-Model — see
  the 2026-06-26 learning about not skipping the V-Model for changes that add
  new user-facing behavior. The 429 response is such a new behavior.
- Out of scope: a shared/persistent rate-limit store (Redis) for multi-instance
  deployments; the app currently runs as a single instance.
