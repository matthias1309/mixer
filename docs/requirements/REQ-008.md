# REQ-008 — Fix `/api/nutrition/ingredients` Endpoint (BUG-009)

**Version**: 1.0
**Date**: 2026-06-15
**Status**: draft
**Epic**: Maintenance — Runtime Defects
**Priority**: P1 (Blocker — endpoint returns 500)
**Owner**: Matthias Bender

---

## 1. Overview

**Description**: `GET /api/nutrition/ingredients` must return the list of
master ingredients (the shared nutrition reference data used for nutrient
calculations) instead of failing with HTTP 500.

**Problem**: The endpoint currently runs
`SELECT * FROM ingredients ORDER BY category, name`. The `ingredients`
table holds per-recipe line items and has no `category` column, so SQLite
raises `SqliteError: no such column: category` and the handler returns a
500 error. Even if the `ORDER BY` were fixed, `ingredients` is still the
wrong table — it would return recipe line-items, not the nutrition
ingredient catalog.

**Business Value**:
- Unblocks any future "ingredient catalog" UI (Phase 2 nutrition features)
  from being built on a working endpoint
- Removes a known 500 error from the API surface

---

## 2. Functional Requirements

### FR-801: Correct Data Source

- **Description**: The endpoint returns rows from `ingredients_master`,
  the table that already backs nutrient lookups elsewhere in the app
  (`src/lib/db/models/ingredientMasterAsync.ts`, recipe nutrient joins in
  `src/lib/db/models/recipe-async.ts`)
- **Acceptance Criteria**:
  - AC-008-01: `GET /api/nutrition/ingredients` returns HTTP 200
  - AC-008-02: The response `data` array contains rows from
    `ingredients_master` (fields include `id`, `name`, `category`,
    `base_unit`, `base_size`, nutrient columns)
  - AC-008-03: Results are ordered by `category`, then `name`

### FR-802: Regression Coverage

- **Description**: A test reproduces the original 500 failure and passes
  after the fix
- **Acceptance Criteria**:
  - AC-008-04: An integration test calls the route handler and asserts a
    200 response with the expected ordering
  - AC-008-05: `npm run lint`, `npx tsc --noEmit`, and `npm test` all pass

---

## 3. Non-Functional Requirements

- **Consistency**: No new dual-database divergence — the query must work
  for the SQLite code path used in production (per CLAUDE.md, the `pg`
  path is migration-tooling only)
- **Scope**: No schema changes; this is a query fix only
