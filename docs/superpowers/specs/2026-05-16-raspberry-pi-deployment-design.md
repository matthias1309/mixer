# Raspberry Pi Deployment Design

**Date:** 2026-05-16  
**Feature:** Complete Docker deployment to Raspberry Pi with PostgreSQL  
**Status:** Design Phase  

---

## Overview

Deploy the Recipe Manager application to a Raspberry Pi with Docker containerization and PostgreSQL database. The deployment must be fully automated via a local script and minimize manual intervention on the PI.

**Scope:**
- Docker Compose setup with Next.js app + PostgreSQL
- App runs on port 3001 (existing service on 3000)
- Automated deployment script for local machine
- One-time manual initialization on PI
- Application connects to PostgreSQL (development uses SQLite)

**Target Location:** `/opt/containers/apps/mixer/` on Raspberry Pi  
**Access:** `ssh dockerhome` from local machine  
**Environment:** Production-grade, Single instance (no scaling)

```
└─────────────────┘             │                      │

┌─────────────────┐             │ │ Docker Compose   │ │
│ deploy-pi.sh    │─ copy ─────→│ │ production.yml   │ │
│ script          │             │ │                  │ │
└─────────────────┘             │ └──────────────────┘ │
                                │                      │
                                │ ┌──────────────────┐ │
                                │ │ mixer-app        │ │
                                │ │ (port 3001)      │ │
                                │ └──────────────────┘ │
                                │                      │
                                │ ┌──────────────────┐ │
                                │ │ mixer-db         │ │
                                │ │ (PostgreSQL)     │ │
                                │ └──────────────────┘ │
                                │                      │
                                │ /data/postgres/      │
                                │ (persistent volume)  │
                                └──────────────────────┘
```

### Container Architecture

**docker-compose.production.yml:**
- **Service 1: mixer-app** (Next.js application)
  - Image: `mixer-app:latest` (built locally, transferred to PI)
  - Port: 3001 (exposed to host)
  - Environment: `NODE_ENV=production`, database URL, secrets
  - Depends on: mixer-db

- **Service 2: mixer-db** (PostgreSQL)
  - Image: `postgres:16-alpine`
  - Port: 5432 (exposed only for debugging/maintenance, not to network)
  - Volumes: `./data/postgres:/var/lib/postgresql/data` (persistent storage)
  - Users/Credentials: Set via environment variables from deploy script

### Data Flow

```
User Browser (port 3001)
         ↓
    mixer-app (container)
         ↓
  PostgreSQL (container, via "mixer-db" DNS)
         ↓
  /opt/containers/apps/mixer/data/postgres/
  (persistent volume on PI disk)
```

## Docker Image (Dockerfile)

### Multi-Stage Build

**Stage 1: Builder**
- Base image: `node:20-alpine`
- Install dependencies: `npm ci`
- Build Next.js: `npm run build`
- Size: ~500MB (discarded after build)

**Stage 2: Runtime**
- Base image: `node:20-alpine`
- Copy only necessary files from builder
- Final size: ~200MB
- Run command: `npm run start` (production Next.js server)
- Expose port: 3001

### Environment Defaults
```dockerfile
ENV NODE_ENV=production
ENV PORT=3001
```

### Database Connection
- Reads `DATABASE_URL` environment variable
- If not set: app won't start (fail fast)
- On PI: injected by docker-compose via deploy script

## Docker Compose (Production Config)

```yaml
version: '3.8'

services:
  mixer-app:
    image: mixer-app:latest
    container_name: mixer-app
    ports:
      - "3001:3001"
    environment:
      DATABASE_URL: postgresql://${DB_USER}:${DB_PASSWORD}@mixer-db:5432/${DB_NAME}
      NODE_ENV: production
      JWT_SECRET: ${JWT_SECRET}
      # Additional env vars as needed
    depends_on:
      mixer-db:
        condition: service_healthy
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3001/api/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  mixer-db:
    image: postgres:16-alpine
    container_name: mixer-db
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - ./data/postgres:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    restart: unless-stopped
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER}"]
      interval: 10s
      timeout: 5s
      retries: 5
```

**Key Features:**
- Variables injected from `.env.production` at runtime
- Health checks ensure services are ready before starting
- `restart: unless-stopped` ensures auto-recovery on crashes
- Database port exposed only locally (5432) for maintenance
- App port exposed to network (3001)

## Deployment Script (deploy-pi.sh)

### Purpose
Automate: build → transfer → start on PI

### Flow

1. **Validation**
   - Check required parameters (JWT secret, DB password)
   - Verify local Docker is running
   - Test SSH connection to PI

2. **Build Docker Image**
   ```bash
   docker build -f Dockerfile -t mixer-app:latest .
   ```

3. **Prepare PI**
   - SSH to PI
   - Create `/opt/containers/apps/mixer/` if missing
   - Stop running containers: `docker compose down`

4. **Transfer Image**
   ```bash
   docker save mixer-app:latest | ssh dockerhome 'docker load'
   ```
   (More efficient than pushing to registry)

5. **Generate & Copy Configuration**
   - Generate `.env.production` with injected secrets
   - Copy `docker-compose.production.yml` to PI
   - Copy `.env.production` to PI (via SCP)

6. **Start Services**
   ```bash
   ssh dockerhome 'cd /opt/containers/apps/mixer && docker compose up -d'
   ```

7. **Verify**
   - Wait 10 seconds for services to initialize
   - Test HTTP request to `http://PI_IP:3001`
   - Print status and logs if failed

### Usage Example

```bash
./scripts/deploy-pi.sh \
  --jwt-secret "your-secure-jwt-key-here" \
  --db-password "secure-postgres-password" \
  --db-user "mixer_user" \
  --db-name "mixer"
```

### Security Considerations

- Secrets passed as command-line arguments (not in shell history when using `read -s`)
- `.env.production` created fresh for each deploy (no caching)
- SSH key-based auth to PI (password-less, configured in `~/.ssh/config`)
- Secrets never written to git

## Initial PI Setup (One-Time Manual Steps)

### Prerequisites
- Docker & Docker Compose already installed on PI ✓
- SSH access via `ssh dockerhome` configured ✓
- Network connectivity between local machine and PI ✓

### Steps

1. **Create directory structure**
   ```bash
   ssh dockerhome
   mkdir -p /opt/containers/apps/mixer/data/postgres
   chmod 755 /opt/containers/apps/mixer
   exit
   ```

2. **Run first deployment**
   ```bash
   ./scripts/deploy-pi.sh --jwt-secret "xxx" --db-password "yyy" ...
   ```
   First run creates all directories, downloads images, initializes database.

3. **Verify deployment**
   ```bash
   curl http://PI_IP:3001
   # Should return home page HTML or redirect to /login
   ```

### That's it
- No manual database setup needed (Docker handles it)
- No manual migration running needed (app handles on startup)
- No configuration files to edit manually

## Database Connection (App Side)

### Changes to Codebase

**File: `src/lib/db/init.ts`**
- Check `process.env.DATABASE_URL`
- If exists: Connect to PostgreSQL via `pg` package
- If not exists: Fall back to SQLite (local development)

**File: `package.json`**
- Add dependency: `"pg": "^8.11.0"` (or latest)

**Connection String Format**
```
postgresql://mixer_user:password@mixer-db:5432/mixer
```

### Local Development (Unchanged)
- No `DATABASE_URL` set → uses SQLite (`.data/app.db`)
- Works as before: `npm run dev`

### Production on PI
- `DATABASE_URL` injected by docker-compose
- App connects to PostgreSQL automatically
- Migrations run on app startup (existing logic)

### Database Initialization
- First run: PostgreSQL creates database automatically (via `POSTGRES_DB` env var)
- Migrations: App runs existing migrations from `src/lib/db/migrations/`
- Fallback: If app crashes before migrations, next `docker compose restart mixer-app` re-runs them

## Configuration Management

### Secrets & Environment Variables

**Local Development**
- `.env.local` (git-ignored, user creates)
- Contains: Dev database URL, test secrets
- Optional: Not needed if using defaults

**Production (PI)**
- `.env.production` (generated & copied by deploy script)
- Contains: Real secrets (JWT, DB password)
- Never committed to git
- Created fresh for each deployment

### Secret Values

```bash
# Provided by user to deploy script
JWT_SECRET=<generated-strong-secret>
DB_USER=mixer_user
DB_PASSWORD=<generated-strong-password>
DB_NAME=mixer

# Used in docker-compose
DATABASE_URL=postgresql://$DB_USER:$DB_PASSWORD@mixer-db:5432/$DB_NAME
```

### How Secrets are Handled

1. User provides as CLI args to deploy script
2. Script creates `.env.production` file locally
3. Script copies file to PI via SCP (over SSH, encrypted)
4. Docker compose reads `.env.production` on PI
5. Variables injected into containers at runtime
6. Original secrets never stored in code or git

## Health Checks & Monitoring

### Built-in Health Checks

**mixer-app health check:**
- Endpoint: `GET /` (or homepage request)
- Interval: 30 seconds
- Docker considers container healthy when endpoint returns successful response (can be implemented as simple `/api/health` if needed)

**mixer-db health check:**
- Command: `pg_isready -U $DB_USER`
- Interval: 10 seconds
- PostgreSQL responds when ready to accept connections

**Docker Compose Dependency:**
- App waits for DB health check before starting
- Prevents connection errors on startup

### Manual Monitoring

```bash
# Check status
ssh dockerhome 'cd /opt/containers/apps/mixer && docker compose ps'

# View logs
ssh dockerhome 'cd /opt/containers/apps/mixer && docker compose logs -f mixer-app'

# Check app response
curl http://PI_IP:3001
```

## Update & Redeploy Process

### Deploying Code Changes

1. **Make changes locally & test**
   ```bash
   npm run dev
   npm run test
   ```

2. **Commit & ready to deploy**
   ```bash
   ./scripts/deploy-pi.sh \
     --jwt-secret "..." \
     --db-password "..." \
     ...
   ```

3. **Script handles the rest:**
   - Rebuilds Docker image with new code
   - Transfers to PI
   - Stops old containers
   - Starts new containers
   - App is down ~30 seconds during transition

4. **Verify**
   ```bash
   curl http://PI_IP:3001
   ```

### Zero Data Loss
- PostgreSQL volume persists across restarts
- Database is never deleted during updates
- Existing recipes/users remain intact

## Troubleshooting

### Common Issues

**Port 3001 already in use**
- Check: `ssh dockerhome 'lsof -i :3001'`
- Solution: Change port in docker-compose or stop other service

**Database connection fails**
- Check: `docker compose logs mixer-app`
- Verify: `docker compose logs mixer-db`
- Solution: Ensure mixer-db is healthy before app starts

**App crashes on startup**
- Check: `docker compose logs mixer-app`
- Likely: Missing environment variables
- Solution: Verify `.env.production` was copied to PI

**Image transfer is slow**
- Docker image size: ~200MB
- Transfer time: Depends on network speed
- Solution: Use local docker daemon caching

## Notes

- **Simplicity First**: No Kubernetes, no service mesh, no load balancing — single-instance deployment on PI
- **KISS Principle**: Minimal setup, maximum clarity
- **No Downtime Tolerance**: App can be offline briefly during updates (acceptable for home lab)
- **Future Scaling**: If needed later, can add Nginx reverse proxy, but not required now
- **Security**: SSH key-based auth, secrets not in git, health checks built-in
